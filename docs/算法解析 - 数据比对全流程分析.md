# 数据比对全流程分析（修正版）

## 前置条件

- **批次大小**：batchSize = 2（为演示方便，实际为 1000）
- **主键类型**：int64
- **读取方式**：`WHERE pk >= ? ORDER BY pk LIMIT batchSize+1`
- **nextKey 选择**：较小的值（修复后）

```

```

| 3 | 5 | {5} | {5} | nil | nil | 5 匹配 delete | source={}, target={} |

```

```
**结论**：✓ 正确报告"无差异"

## 附录：缺失数据对批次的影响分析

### 问题：target 缺失 2 条数据，总批次增加多少？

### 数据分布
```
源表：1-10000（共 10000 行）
目标表：缺 500 和 600（共 9998 行）
batchSize = 1000
```

### 第 1 批：currentKey = 1

| 表 | 查询 | 返回数据 | nextKey |
|---|------|----------|---------|
| source | `SELECT * FROM source WHERE pk >= 1 LIMIT 1001` | 1-1000（1000 行），第 1001 行=1001 | **1001** |
| target | `SELECT * FROM target WHERE pk >= 1 LIMIT 1001` | 跳过 500,600 取 1001 行 | **1003** |

**target nextKey 详细计算**：

| 范围 | 主键数量 | 累计行数 |
|------|----------|----------|
| 1-499 | 499 行 | 499 |
| 501-599 | 99 行 | 598 |
| 601-1000 | 400 行 | 998 |
| 1001 | 1 行 | 999 |
| 1002 | 1 行 | 1000 |
| 1003 | 1 行（第 1001 行） | 1001 |

```
targetNext = 1003 ✓
```

**nextKey 选择**：
```
nextKey = min(1001, 1003) = 1001
```

### 第 2 批：currentKey = 1001

| 表 | 查询 | 返回数据 | nextKey |
|---|------|----------|---------|
| source | `SELECT * FROM source WHERE pk >= 1001 LIMIT 1001` | 1001-2000（1000 行） | 2001 |
| target | `SELECT * FROM target WHERE pk >= 1001 LIMIT 1001` | 1001-2000（1000 行） | 2001 |

**说明**：1001-2000 范围内 target 没有缺失，所以两表完全对齐。

### 第 3-10 批

从第 2 批开始，两表完全对齐，无需额外批次。

### 结论

| 指标 | 结果 |
|------|------|
| **总批次** | 10 次 |
| **比正常增加** | +0 次 |
| **原因** | 缺失数据在第 1 批内部被"消化"，后续批次对齐 |

---

## nextKey 计算公式

```
当 target 缺失 k 条数据时：

targetNext = sourceNext + k  （假设缺失都在当前批次内）

nextKey = min(sourceNext, sourceNext + k) = sourceNext
```

**关键观察**：
- 缺失数据会导致 targetNext 偏移
- 但选择较小的 nextKey 确保不会跳过任何记录
- 缺失数据在后续批次中会被"追平"
