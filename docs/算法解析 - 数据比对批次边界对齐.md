# 算法解析：数据比对批次边界对齐问题

## 问题背景

在数据库全量数据比对中，当源表和目标表的主键分布不同时，原有的批次比对逻辑会产生误判，将两个表都存在的记录错误地标记为"新增"或"删除"。

### 实际案例

比对 `core_license_photo_en` 表时：
- **实际情况**：源表和目标表都有 `photo_id=108624`
- **错误报告**：108624 被标记为"新增行"

---

## 问题根源分析

### 1. 批次边界不对齐

当两个表的数据分布不同时，同一批次读取的主键范围可能不完全重叠。

**示例**：
```
源表主键：[1, 2, 3, 5]
目标表主键：[1, 2, 3, 4, 5]
batchSize = 2
```

### 2. nextKey 选择错误

原代码选择**较大**的 `nextKey`：
```go
if sourceNext[0].(int64) > targetNext[0].(int64) {
    nextKey = sourceNext[0]  // 选择较大的
} else {
    nextKey = targetNext[0]
}
```

**问题**：如果 `sourceNext=5`，`targetNext=4`，选择 `nextKey=5`，则目标表的主键 4 会被跳过！

### 3. 过早标记"删除/新增"

原代码在每批次结束后立即标记：
```go
for _, srcRow := range sourceMap {
    diff.Removed = append(diff.Removed, srcRow)  // ❌ 过早标记
}
for _, tgtRow := range targetMap {
    diff.Added = append(diff.Added, tgtRow)
}
```

**问题**：此时这些记录可能只是还没遇到匹配项，而不是真的不存在于另一个表。

---

## 错误流程演示

### 原代码执行流程

| 批次 | currentKey | sourceBatch | targetBatch | 比对后剩余 | 误判结果 |
|------|------------|-------------|-------------|------------|----------|
| 1 | 1 | {1, 2} | {1, 2} | 空 | - |
| 2 | 3 | {3, 5} | {3, 4} | source={5}, target={4} | 5→删除 ❌, 4→新增 ❌ |

**第 2 批结束后**，原代码立即将 5 标记为"删除"、4 标记为"新增"，但它们实际上都有匹配项（在后续批次中）。

---

## 修复方案

### 核心思想

1. **累积未匹配记录**：使用全局 map 累积未匹配的记录，而不是每批次新建局部 map
2. **选择较小的 nextKey**：确保不会跳过任何记录
3. **延迟标记**：只有在所有批次处理完成后，才将剩余记录标记为"删除/新增"

### 修复后代码

```go
// 1. 全局 map 累积未匹配记录
sourceRemaining := make(map[string]map[string]interface{})
targetRemaining := make(map[string]map[string]interface{})

for {
    // 2. 读取当前批次
    sourceBatch, sourceNext, _ := fetchDataByKeyRange(...)
    targetBatch, targetNext, _ := fetchDataByKeyRange(...)

    // 3. 合并到全局 remaining（而不是新建局部 map）
    sourceMap := sourceRemaining
    targetMap := targetRemaining
    for _, row := range sourceBatch {
        key := buildKey(row, pkCols)
        sourceMap[key] = row
    }
    for _, row := range targetBatch {
        key := buildKey(row, pkCols)
        targetMap[key] = row
    }

    // 4. 比对，只 delete 匹配的
    for key, srcRow := range sourceMap {
        if tgtRow, exists := targetMap[key]; exists {
            // 匹配，进行详细比对
            delete(sourceMap, key)
            delete(targetMap, key)
        }
        // 不匹配的保留在 map 中，进入下一批次
    }

    // 5. 选择较小的 nextKey，确保不会遗漏
    if sourceNext != nil && targetNext != nil {
        if sourceNext[0].(int64) < targetNext[0].(int64) {
            nextKey = sourceNext[0]  // ← 选择较小的
        } else {
            nextKey = targetNext[0]
        }
    } else if sourceNext != nil {
        nextKey = sourceNext[0]
    } else if targetNext != nil {
        nextKey = targetNext[0]
    } else {
        break  // 两个表都没有更多数据
    }
}

// 6. 循环结束后，remaining 中的才是真正的"删除/新增"
for _, srcRow := range sourceRemaining {
    diff.Removed = append(diff.Removed, srcRow)
}
for _, tgtRow := range targetRemaining {
    diff.Added = append(diff.Added, tgtRow)
}
```

## 修复后流程演示

### 修复后执行流程

| 批次 | currentKey | sourceBatch | targetBatch | remaining 比对后 |
|------|------------|-------------|-------------|------------------|
| 1 | 1 | {1, 2} | {1, 2} | 1,2 匹配 → 空 |
| 2 | 3 | {3, 5} | {3, 4} | 3 匹配 → source={5}, target={4} |
| 3 | 4 | {5} | {4, 5} | 5 匹配 → source={}, target={4} |
| 结束 | - | - | - | **最终**：target={4} → 4 是新增 ✓ |

---

## 关键设计决策

### 1. 为什么使用全局 remaining map？

**问题**：局部 map 在每批次结束后就被丢弃，未匹配的记录失去后续匹配机会。

**解决**：全局 map 让未匹配的记录累积，在后续批次中继续参与比对。

### 2. nextKey 为什么选择较小的？

| 选择 | sourceNext=5, targetNext=4 | 结果 |
|------|---------------------------|------|
| 较大值 | nextKey=5 | 目标表的 4 被跳过 ❌ |
| 较小值 | nextKey=4 | 两个表都从 4 开始，4 有机会匹配 ✓ |

**代价**：目标表的 4 会被重复读取，但 `map[key] = row` 会覆盖，不会造成重复报告。

### 3. 为什么在循环结束后处理？

**所有批次的主循环结束后**（`for { ... }` 的 `break` 之后），才能确保：
- 两个表的所有记录都已读取
- remaining 中的记录**真的**在另一个表中不存在

## 复杂度分析

| 指标 | 原代码 | 修复后 |
|------|--------|--------|
| 时间复杂度 | O(n) | O(n)，但有少量重复读取 |
| 空间复杂度 | O(batchSize) | O(n) 最坏情况 |
| 准确性 | ❌ 批次边界不对齐时误判 | ✓ 正确处理所有情况 |

---

## 适用场景

此修复适用于以下场景：
- 两个表的主键分布不同（有插入/删除）
- 批次大小固定（如 1000 行/批）
- 按主键范围分批读取（`WHERE pk >= ? ORDER BY pk LIMIT N`）
